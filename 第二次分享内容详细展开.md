## 第一部分：产品诞生背景（5分钟）

### 1.1 业务痛点
**机构客户的现实困境：**
- 机构客户需要识别大量管理人的单据（交易确认单、分红确认单等）
- 一般机构都有几十家管理人，不同管理人、不同业务类型的单据模板各异
- 传统方式：每个客户、每个管理人、每个单据类型都要定制开发
- **问题**：周期长、维护难、复用性差、成本高

**高频场景：**
- **运营日常需要处理各种管理人的单据识别**
- **场景特征**：
  - 批量处理、格式多样、时效要求高
  - 不同管理人的单据模板都不一样
  - 不同业务类型的单据格式差异大

**核心痛点：**
- 不同管理人的单据模板各异
- 人工识别和录入投入成本巨大
- 容易出错，准确率难以保障
- 无法批量处理，效率低下

## 第二部分：我们做了什么产品（5分钟）

### 2.1 产品介绍
我们做了一款**蜂鸟智能文档助手**（基于Dify平台的AI智能识别工具）。

**产品定位：**
- **通用场景的智能文档识别工具**：不仅面向机构客户，B端和C端客户都可以使用
- 支持多管理人、多业务类型的单据识别
- 零代码配置，用户自主配置字段映射关系
- 适应不同客户类型的需求：机构客户的批量处理、B端客户的业务场景、C端客户的日常使用

### 2.2 核心能力
**技术方案：**
- 基于Dify平台的零代码方案
- PaddleOCR + 本地大模型qwen-72b（协同长研团队集成）
- 本地化部署，避免数据外发
- 支持9种文件格式（PDF、DOC、DOCX、XLS、XLSX、PPT、PPTX、JPG、PNG）
- MinerU工具集成（协同长研团队完成）
- Deepseek本地大模型（协同长研团队集成）

**核心功能：**
- OCR识别：自动识别单据内容
- 多模式选择：长/快思考双模式，适应不同场景
- 批量处理：支持批量并行处理
- 邮件通知：异步获取处理结果
- 零代码配置：用户自定义字段映射

### 2.3 产品成果
**数据验证：**
- 半年累计处理：约3000份文档
- 识别准确率：
  - 简单文档：98%
  - 复杂文档：94%
  - 整体平均：96.5%
- 性能指标：
  - 处理速度提升12倍
  - 单日处理能力提升10倍

**应用形式：**
- 本地化部署（已成功在某全国性股份制银行理财子公司落地）
- 标准API/MCP工具（支持嵌入核心系统）
- 网页版（手机号登录，即开即用）

## 第四部分：心路历程（27分钟）

接下来我会按照时间线，分享每个阶段我们做了什么，遇到了什么问题，如何解决的，以及心得体会。

### 4.1 第一阶段：从需求到想法验证（5分钟）

**我们做了什么：**

**第一步：遇到客户需求问题**
- 机构客户提出需要识别大量管理人的单据
- 分析传统方式的痛点：定制开发成本高、周期长
- 开始探索AI解决方案的可能性

**第二步：发现Dify平台**
- 公司正好在推广Dify平台
- Dify提供了零代码搭建AI工作流的能力
- 意识到这个平台可能适合快速验证想法

**第三步：借助Dify初步验证想法**
- 在Dify平台上搭建初步的工作流
- 验证OCR + 大模型结合的技术可行性
- 测试基本流程：上传文件 → OCR识别 → 大模型提取 → 输出结果
- 用简单单据样本进行初步验证

**第四步：借助Cursor跑通PaddleOCR效果**
- 使用Cursor等AI编程工具实际编写和运行代码
- 导入PaddleOCR库，配置识别参数
- 用不同类型的单据样本测试识别效果
- **对比测试**：
  - 传统OCR vs PaddleOCR的效果差异
  - 不同格式文件（JPG、PDF、扫描件）的识别准确率
  - 表格内容的识别准确度
- 记录识别效果、准确率等关键指标

**优化尝试：使用PaddleOCR坐标和置信度提升准确度**
- 当时还没有集成MinerU，为了提升识别准确度，尝试利用PaddleOCR输出的坐标和置信度信息
- 通过坐标信息定位文本位置，通过置信度筛选高质量识别结果
- **测试发现**：
  - 在本地大模型（qwen-72b）的基础上，效果一般，提升不明显
  - 在云端更强模型的基础上，效果不错，能够有效提升识别准确率
- 这个尝试帮助我们理解了不同模型能力对OCR后处理效果的影响，也为后续选择本地部署方案提供了参考

**遇到什么问题：**
- 不确定AI能否准确理解单据结构
- 不知道OCR + 大模型结合是否可行
- 需要验证不同技术方案的效果差异

**怎么解决的：**
- 通过Dify快速搭建demo，验证技术路线可行性
- 使用Cursor实际运行代码，对比测试不同方案
- 用真实单据样本进行验证，建立对技术能力的认知

**心得体会：**
1. **先做demo验证可行性**：产品经理的快速验证思维，避免大投入失败
2. **AI工具让验证成本大幅下降**：过去需要研发配合几周，现在产品经理可以独立验证
3. **多动手，多测试**：通过实际运行代码，建立对技术方案的直观认知

### 4.2 第二阶段：打造MVP（6分钟）

**我们做了什么：**

**第五步：收集需求打造MVP**
- **需求收集**：
  - 与内部运营团队深入沟通，了解真实业务场景
  - 收集不同管理人的实际单据样本
  - 理解不同类型单据的识别难点和特殊要求
- **MVP设计**：
  - 基于真实需求设计MVP功能清单
  - 确定核心功能：单据上传、字段识别、结果导出
  - 设计零代码配置方案，支持用户自定义字段映射
- **MVP搭建**：
  - 在Dify平台上搭建完整工作流
  - 协同长研团队集成PaddleOCR和本地大模型qwen-72b
  - 实现字段映射配置功能
- **格式验证结果**：
  - PDF格式（文本） → 识别准确 ✓
  - PDF格式（文本+表格） → 识别效果好 ✓
  - PDF格式（表格） → 表格识别准 ✓
  - 扫描件/图片格式（纯文本） → 识别准确 ✓
  - **图片格式的表格** → 效果一般 ⚠️
  - 原始单据（JPG换行场景） → AI识别效果 ✓

**验证总结：** 文本和图片效果都可以，但涉及到图片格式的表格，效果就一般。这也是后来为什么要集成MinerU工具的原因之一。

**遇到什么问题：**

**① 测试不同类型单据的效果（基于本地部署的大模型）**
- 必须基于本地部署的大模型进行测试，确保符合金融行业的数据安全要求
- 测试不同管理人的各类单据识别效果
- 验证不同业务类型单据模板的处理能力
- 这个过程需要大量的样本测试和效果验证

**② 调优提示词**
- 不断优化给大模型的提示词（Prompt）
- 确保AI能够准确理解单据结构和提取需求
- 适配不同单据模板的差异化需求

**③ 确保灵活可配置，符合内外部场景**
- 设计零代码配置方案，让用户能够自主配置字段映射
- 确保方案既适用于内部使用场景，也适用于外部客户场景
- 在灵活性和易用性之间找到平衡
- 降低使用门槛，提升产品适用性

**④ 在Dify工作流兼容不同的业务场景、不同的单据模板**
- 设计通用的工作流架构，确保可扩展性
- 兼容不同管理人的单据模板差异
- 确保未来的可移植性，方便在不同环境下部署
- 为产品化复制推广打好基础

**怎么解决的：**
- 逐步摸索Dify平台，从简单到复杂，从组件到代码
- **Dify平台使用的组件**：
  - **迭代组件**：用于多页文档的逐页处理
  - **代码组件**：实现复杂的逻辑处理和数据处理
  - **模板转换组件**：支持不同单据模板的结构化转换
  - 其他组件组合，构建完整的识别工作流
- 借助AI工具（如Cursor）帮助理解和调试
- 与长研团队紧密协作，共同打磨功能
- 大量测试不同单据样本，不断优化提示词和配置
- 在Dify工作流中设计通用架构，支持不同场景

**心得体会：**
1. **产品经理可以深度参与技术实现**：这个产品的完整Dify流程完全由产品经理侧搭建，体现了AI时代下产品经理的能力边界正在扩展
2. **产品经理需要多动手**：即使不懂代码，也要熟悉工具边界和成本
3. **借助AI工具降低门槛**：使用Cursor等工具，不需要精通代码细节，但要理解代码的逻辑和步骤
4. **产研协作很重要**：虽然Dify流程由产品经理搭建，但技术组件的集成（如PaddleOCR、大模型等）需要协同研发团队完成，工程问题交给研发团队
5. **场景细分很重要**：在金融领域，必须考虑AI准确性问题，选择适合的场景

### 4.3 第三阶段：内部打磨（8分钟）

**我们做了什么：**

**第六步：内部运营持续试用，持续打磨**
- MVP版本先给内部运营团队使用
- 收集真实使用反馈和问题
- **打磨功能场景丰富度**：从单一场景扩展到多管理人、多业务类型
- **打磨操作体验**：优化用户界面、流程体验、结果呈现
- 基于反馈持续迭代优化（V1.0 → V1.5，6个版本迭代）

**产品功能迭代机制：**
在这个过程中，我建立了一套产品功能迭代的方法：
- **不定期查看后台日志**：主动监控产品运行状态，发现潜在问题
- **分析失败案例**：重点关注哪些用户失败了，失败原因是什么
- **找客户了解情况**：主动联系用户，深入了解失败的具体场景和原因
- **要数据自己测试**：向客户要失败的单据样本，自己实际测试效果
- **看结果再迭代**：基于测试结果分析问题，提出改进方案，持续迭代

这套机制让我能够主动发现问题，而不是被动等待用户反馈，从而更快速地响应和解决问题。

**第一次迭代（V1.1 - 1个月后）：**
- **用户反馈**：多页表格识别准确率不足
- **问题分析**：大批量同类数据提取质量不稳定
- **解决方案**：引入迭代式并行处理
  - 逐页并行处理
  - 结果组装优化
- **效果**：多页表格识别准确率有所提升

**第二次迭代（V1.2 - 2个月后）：**
- **用户需求**：需要兼容更多识别场景
- **解决方案**：引入双模式识别（协同长研团队完成）
  - 长思考模式：Deepseek R1本地大模型，复杂场景，准确但慢
  - 快思考模式：Qwen2.5-32b，简单场景，快速
- **亮点**：支持指定页码识别（例如："1-3, 5, 7-9"）
- **效果**：用户可根据场景选择最优模式

**第三次迭代（V1.3 - 3个月后）：**
- **用户痛点**：
  - 等待焦虑，不知道处理进度
  - 识别结果如果没有及时保存就关闭了网页，尤其是在大批量文件识别的时候，容易丢失结果
- **解决方案**：
  - 新增进度条：可视化AI生成时间，解决等待焦虑
  - 新增邮件通知：异步获取结果，即使关闭网页也能通过邮件获取识别结果，避免大批量处理时结果丢失

**关键里程碑：协同长研集成MinerU（V1.4）**
- 协同长研团队成功集成MinerU工具
- **实现PDF版式识别能力**：解决了传统PaddleOCR对图片格式表格识别准确度不高的问题
- **支持复杂表格识别**：表格内容有换行也能完整识别
- **扩展文件格式支持**：新增doc格式支持
- **提升识别准确度**：特别是表格和版式复杂文档的识别效果有所改善

  - 准确度和速度并不冲突，可以通过技术优化同时提升
- **优化识别路径**：

**第十步：跑通对应的MCP调用**
- 打造标准API/MCP工具
- 支持以MCP工具形式嵌入其他系统
- 实现AI能力的标准化输出和调用

**遇到什么问题：**

**AI产品的准确性挑战：**
- AI产品始终有个绕不过去的问题，就是幻觉和准确性问题
- 在金融机构中，对交易、数据的准确度必须是100%，99.99%都不可接受
- 必须严格筛选场景，不是所有场景都适合直接应用AI能力

**怎么解决的：**

**场景细分的思考框架：**
在选择AI应用场景时，我们进行了深入细分：
- **哪些场景可以集成AI**：准确率要求相对宽松，或者有容错机制的场景
- **哪些场景不行**：对准确性要求极高，不允许任何误差的场景
- **不行场景中的哪些环节可以集成**：即使整体场景不行，某些环节可能可以应用
- **不行场景为什么不行**：明确技术边界和风险点
- **能不能通过复核环节解决**：考虑增加人工复核环节，确保准确性
- **复核成本是否可接受**：复核成本要低于原本场景的投入成本，才有应用价值

**这个项目的场景选择：**
我们选择的单据识别场景属于**可以增加复核环节**的场景：
- **增加复核环节**：AI识别结果经过人工复核确认，确保100%准确性
- **成本依然更优**：AI预处理 + 人工复核的成本，远低于传统完全人工处理的成本
- **对AI结果负责**：通过复核机制，对AI结果承担责任，符合金融机构的合规要求
- **效率提升**：即使增加复核环节，整体效率仍高于纯人工方式

**产品推广路径：**
虽然需求来自客户，但产品推广采取了"内部打磨 → 客户落地"的路径：
- **MVP版本先给内部用**：先在公司内部使用，进行真实场景验证
- **持续打磨优化**：基于内部使用反馈，持续迭代完善（V1.0 → V1.5，6个版本迭代）
- **再向客户推广落地**：产品成熟后，再向客户推广，支持本地化部署

**心得体会：**

**产品化的完整路径：**
1. **快速验证**：MVP快速验证可行性，降低试错成本
2. **找到高频场景**：聚焦核心业务场景，解决真实痛点
3. **解决痛点问题**：针对高频场景的核心痛点，提供有效解决方案
4. **持续迭代打磨**：基于用户反馈持续优化，持续完善产品
5. **产品化复制推广**：将成熟方案标准化，支持机构客户本地部署

**关键体会：**
1. **需求驱动，而非技术驱动**
2. **内部打磨降低风险**：在可控环境下充分验证产品能力
3. **场景选择很关键**：在金融领域，必须考虑准确性和复核机制
4. **从工具到产品，从产品到平台**
